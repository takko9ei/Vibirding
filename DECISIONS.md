# DECISIONS — 取舍记录

> 每个取舍记三行：**决定 / 为什么 / 代价或取舍**。面试逐字稿用。

## read_log 只是"个人弱先验"，季节/分布核验另设权威工具
- **决定**：`read_log` 的职责收敛为"查我自己的历史观测"（个人记录 / 弱先验）；"某地某季是否合理出现某种鸟"这类权威核验，留给独立的鸟种分布工具 range_check（数据源 eBird，已升为正式切片 S3，见 §7/§10）。
- **为什么**：个人日志不代表生物地理事实——没记录过 ≠ 那儿没有，记录过也只是弱信号；权威物候/分布数据才是正主。v1 实测中 read_log 对季节判断几乎没提供权威价值，模型只能回退到自身知识。
- **代价/取舍**：v1 暂不接 eBird（控成本与范围），季节核验先靠模型知识 + read_log 弱先验；权威分布工具 range_check 已排为正式切片 S3；核验子 agent 仍作为后续进阶（§11）。

## range_check 用范式B（先取清单、模型再匹配），不用范式A（先猜种、工具核验单种）
- **决定**：`range_check(place, date)` 返回**该地近期实际记录的物种清单**，模型在清单内挑与外形描述匹配的种；而不是"模型先猜一个种、工具去回答这个种在不在"。
- **为什么**：清单把"开放式鸟类学知识题"收敛成"闭卷选择题"——模型只能从真实出现过的种里选，更准、更难凭空编造；eBird recent 一次就拿回全量，省往返。
- **代价/取舍**：清单可能很长（用 `back` 天 + 展示截断收窄）；"当季"用 recent 作代理，吃不了任意历史日期；清单外的罕见种会被漏（个人速记重常见种，可接受）。

## 地名→坐标用预存坐标表，不接地理编码 API
- **决定**：地名→经纬度用手维护的"常去观测点预存表"（`tools/locations.py`，按标准名 exact-match），命不中即降级（返回引导文本、不报错），不接 geocoding 服务。
- **为什么**：个人观鸟点就那么几个、低频新增；预存表零依赖、零成本、零延迟、坐标可控；接 geocoding 要多一个 key/网络/歧义消解，为个人级用量不划算。
- **代价/取舍**：命不中只能降级到纯知识推断；换城市/加新点要手动加一行；不具通用性（v1 不需要）。

## range_check 是主 agent 调的普通工具，不是核验子 agent
- **决定**：range_check 做成主 agent 直接调用的只读工具（注册→校验→执行→归一化），不拆成独立的"核验子 agent"。
- **为什么**：它的职责只是"取一份数据"——单步、无状态、不需要自己规划或多轮推理；工具契约已足够，子 agent 会凭空加编排/上下文复杂度，违背"砍到个人级"的右尺寸原则。
- **代价/取舍**：把 bird_id + range_check + read_log 的"二次合理性交叉判断"暂压在主 agent 的 prompt 里；真要做多源核验时再升级为子 agent（§11 进阶），现在不背这税。
